# Upgrading Vulse

> **The rule:** scaffold once, update packages forever. Migrate user files
> only with explicit codemods or documented manual steps.

Vulse follows the same model as Astro, Next.js, and similar frameworks:
the engine lives in `node_modules`, and your project owns its pages,
config, integrations, content, and components. **Never edit files inside
`node_modules/@vulse/*`.** Those changes will be wiped on the next install.

## The package boundary

| Package           | Lives in       | You edit it?           |
| ----------------- | -------------- | ---------------------- |
| `@vulse/core`     | `node_modules` | No                     |
| `@vulse/auth`     | `node_modules` | No                     |
| `@vulse/db`       | `node_modules` | No                     |
| `@vulse/host`     | `node_modules` | No                     |
| `@vulse/image`    | `node_modules` | No                     |
| `@vulse/renderer` | `node_modules` | No                     |
| `@vulse/admin`    | `node_modules` | No                     |
| `vulse.config.ts` | your project   | Yes                    |
| `blueprints/`     | your project   | Yes                    |
| `modules/`        | your project   | Yes (this is the extension point) |
| `server/server.ts`| your project   | Yes (rare — composition only) |
| `src/main.ts`     | your project   | Yes                    |

If you need behaviour change inside Vulse and there is no extension point,
**open an issue**. Forking the packages locally will block your upgrades.

## The upgrade command

```sh
pnpm up "@vulse/*"
```

That's the normal path. The `@vulse/*` packages are *linked* (see
`.changeset/config.json`) so they release together with matching versions —
bumping them in lockstep keeps things consistent.

After upgrading, check `node_modules/@vulse/host` works against your
current `server/server.ts`:

```sh
pnpm dev
pnpm build
```

If `pnpm dev` boots and a smoke test passes, you are done.

## Reading release notes

Every release has a `CHANGELOG.md` inside each package, generated by
changesets. A patch release should never require code changes from you.
A minor release may add new extension points; you can ignore them or
adopt them on your schedule. A major release may rename callbacks or
change a function signature — those changes are documented under
**Breaking changes** in the changelog and in the per-version upgrade
notes below.

## Per-version upgrade notes

### 0.x → 0.x

The 0.x line is pre-stable. Expect occasional breakage at minor bumps until
the architecture settles. Each release listing here will name the file or
function that changed and the one-line edit you need to make.

### Headless pivot (removal of `@vulse/site`)

The `@vulse/site` Vue 3 SSR runtime has been **removed**. Vulse is now a
headless CMS: it ships the admin SPA and the HTTP API only. Your
public-facing site moves out of the Vulse project into a separate
frontend (Astro, Next, SvelteKit, Nuxt, …) that fetches content from
`/api/public/*`.

**If your project used `@vulse/site`:**

1. Remove `@vulse/site` from `package.json`.
2. Delete `vite.config.site.ts` and the `vite build -c vite.config.site.ts`
   step from your `build` script.
3. Remove the `site` field from `vulse.config.ts` and the
   `import type { SiteConfig } from '@vulse/site/server'` line.
4. In `server/server.ts`, drop the `SiteConfig` / `SiteRouteOverrides`
   imports, the `siteStaticRoot` / `siteConfig` / `siteRoutes` locals, the
   `site:` field from `handlerOpts`, and the `{ root: siteStaticRoot, ... }`
   entry from `createNodeServer`. The `buildHandlers` result no longer
   includes a `site` listener.
5. Move your `site/pages/*.vue` files into a new Astro / Next / SvelteKit
   project alongside Vulse and rewrite them to fetch from
   `/api/public/collections/:handle` and `/api/public/globals`. See
   [frontend-foundation.md](./frontend-foundation.md) for the recipe.
6. The `@vulse/renderer` package is still published and can be used from
   any Vue-based frontend that wants to render Vulse block content.

The HTTP API contract (`/api/public/*`, `/_vulse/img/*`,
`/api/blueprints`, auth, drafts, preview tokens) is **unchanged**. Only
the bundled Vue frontend is gone.

## What to do if your `server.ts` falls behind

The `server/server.ts` file in your project is a composition of
`@vulse/host` helpers. New `@vulse/host` versions may add helpers that
let you simplify it. **You are never required to adopt them.** Your
existing composition will keep working unless a release note explicitly
calls out a breaking change.

If you want to refresh `server.ts` from the current `create-vulse`
template, the easiest path is:

```sh
pnpm dlx create-vulse /tmp/vulse-ref
diff /tmp/vulse-ref/server/server.ts server/server.ts
```

Cherry-pick whatever you want.

## When extension points are not enough

If you find yourself wanting to monkey-patch a package, please open a
discussion first. The fix usually belongs in:

- A new event on the bus (see [docs/architecture.md](./architecture.md#1-the-event-bus))
- A new option on an existing helper
- A new module hook

These are easy to add to Vulse without breaking anyone, and they
keep your project upgradeable.
